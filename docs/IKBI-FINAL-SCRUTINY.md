## Residual Gaps (not blockers, but worth knowing)

### R1 [LOW] — `isStubScript` doesn't catch `npx echo pass`

A script like `"test": "npx echo pass"` or `"test": "node -e process.exit(0)"` bypasses stub detection because `npx` and `node` are not in the allowed no-op patterns. The `echo` or `exit 0` is there but wrapped in a real binary invocation.

**Risk:** Low. These patterns are rare in real packages. The operator opt-in (`IKBI_VERIFICATION_LADDER_TRUST_TRIVIAL_SCRIPTS=true`) exists for legitimate trivial scripts.

### R2 [LOW] — No minimum check timeout

`resolveCheckTimeoutMs()` accepts any positive integer. `IKBI_CHECK_TIMEOUT_MS=1` would cause every governed-exec check to timeout at 1ms. This is a denial-of-service vector (operator can't get any check to pass). Not a false-green, but worth a floor.

**Risk:** Low. Requires operator to deliberately set a nonsensical value.

### R3 [LOW] — Binary TS files produce empty import graphs

If a `.ts` file contains binary content (corrupted by git, generated by a codegen tool that outputs binary-looking data), `buf.toString("utf8")` produces garbage. The import extractor regex would find no imports, silently producing an empty import graph for that file.

**Risk:** Extremely low. Requires a corrupted or malicious `.ts` file. The verifier's script-integrity guard would catch package.json-level manipulation, and the scout would read garbage context, but the builder might operate on incomplete dependency information.

### R4 [MEDIUM] — No end-to-end test with a real, large monorepo

All tests use small fixtures. The system has never been tested against a 10,000-file monorepo with 50 packages. The caps (maxFiles=200000, maxImpactFiles=2000, MAX_FILES_SCANNED=40) are theoretically correct but empirically unverified at scale.

**Risk:** Medium. The logic is sound but untested at production scale.

### R5 [LOW] — The `truncated` flag on retrieval is not surfaced to the operator

The project-index sets `truncated: true` when `maxFiles` is exceeded, and the ladder escalates to full. But the operator never sees the `truncated` flag — it's buried in internal data structures. A busy operator might not notice that 40% of the repo was skipped because of the file cap.

**Risk:** Low (correctness is preserved by escalation), but operator overconfidence risk.
